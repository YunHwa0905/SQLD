데이터 모델과 성능

성능데이터모델링

✔ 성능 데이터모델링 : 데이터베이스 성능 향상을 목적으로 설계단계의 데이터 모델링 때부터 성능과 관련된 사항이 데이터모델링에 반영될 수 있도록 하는 것
1) 데이터의 증가가 빠를수록 성능저하에 따른 성능개선비용은 증가
2) 데이터모델은 성능을 튜닝하면서 변경이 될 수 있음
3) 분석/설계단계에서 성능을 고려한 데이터모델링을 수행할 경우 성능 저하에 따른 Rework 비용을 최소화 할 수 있는 기회를 가짐

✔ 성능을 고려한 데이터 모델링의 순서
1) 데이터 모델링을 할 때 정규화를 정확하게 수행
2) 데이터베이스 용량 산정 수행
3) 데이터베이스에 발생되는 트랜잭션의 유형을 파악
4) 용량과 트랜잭션의 유형에 따라 반정규화를 수행
5) 이력모델의 조정, PK/FK조정, 슈퍼타입/서브타입 조정 등을 수행
6) 성능관점에서 데이터 모델을 검증

2. 정규화

✔ 정규화(Normalization): 데이터 중복 최소화 & 이상현상 제거
1) 1NF(제1정규형) : 컬럼의 원자성 보장
   => 중복된 값 분리
2) 2NF(제2정규형) : 부분적 함수 종속 제거
   => 키 일부에만 종속되는 문제 해결
3) 3NF(제3정규형) : 이행적 함수 종속 제거
   => 컬럼 간 종속성 제거
4) BCNF(보이스-코드 정규화) : 모든 결정자가 후보 키가 되도록
   => 이상현상 제거 강화
5) 4NF(제4정규형) : 다치 종속 제거
6) 5NF(제5정규형) : 조인 종속성 이용

3. 반정규화

✔ 반정규화(Denormalization)
1) 성능 향상을 위해 정규화된 데이터 모델을 일부러 중복 허용
2) 지나친 정규화는 JOIN 연산 증가 -> 성능 저하
3) 중복 테이블, 중복 컬럼, 통합 테이블 등을 활용하여 성능 개선
   ✔ 칼럼에 대한 반정규화 기법
1) 중복칼럼 추가 : 조인감소를 위해 여러 테이블에 동일한 칼럼을 갖도록 함
2) 파생칼럼 추가 : 조회 성능을 우수하게 하기 위해 미리 계산된 칼럼을 갖도록 함
3) 이력테이블에 기능 칼럼 추가 : 최신값을 처리하는 이력의 특성을 고려하여 기능성 칼럼 				추가

✔ 다른 방법유도 검토
1) 뷰 테이블 : 지나치게 많은 조인이 걸려 데이터를 조회하는 작업이 기술적으로 어려운 		경우
2) 클러스터링 적용 : 대량의 데이터처리나 부분처리에 의해 성능이 저하되는 경우
3) 인덱스의 조정 : 대량의 데이터처리나 부분처리에 의해 성능이 저하되는 경우
4) 파티셔닝 : 대량의 데이터는 Primary Key의 성격에 따라 부분적인 테이블로 분리
4) 응용애플리케이션 : 로직을 구사하는 방법 변경

✔ 칼럼수가 많은 테이블
=> 로우체이닝이 발생할 정도로 한 테이블에 많은 칼럼들이 존재할 경우 조회성능저하가 발생할 수 있음. 트랜잭션이 접근하는 칼럼유형을 분석하여 1:1로 테이블을 분리하면 디스크 I/O가 줄어들어 조회 성능을 향상시킬 수 있음

✔반정규화 적용
1) 테이블 반정규화
2) 속성의 반정규화
3) 관계의 반정규화

4. 인덱스와 성능 최적화

✔ 인덱스(Index) 개념
1) 검색 속도 향상을 위해 특정 칼럼에 색인을 생성하는 기법
2) B-Tree 인덱스, 해시 인덱스 등 다양한 방식 존재

✔ 인덱스 성능 고려 사항
1) 클러스터드 인덱스 : 물리적 데이터 정렬 => PK가 기본적인 클러스터드
2) 넌클러스터드 인덱스 : 데이터와 분리된 색인 => 검색 속도 향상

✔ 인덱스 주의할 점
1) 너무 많은 인덱스는 오히려 성능 저하 (INSERT/UPDATE 비용 증가)
2) 조회 빈도가 높은 컬럼에만 설정

5. 관계형 모델과 키의 종류

✔ 키(Key) 개념 정리
1) 슈퍼키 : 튜플을 유일하게 식별하는 키 / 여러 속성 포함 가능
2) 후보키 : 최소 속성으로 유일성 유지 / NULL 불가
3) 기본키(PK) : 테이블에서 한 개만 존재 / Not NULL & Unique
4) 대체키 : 후보키 중 선택되지 않은 키 / 후보키 중 PK가 아닌 것
5) 외래키(FK) : 다른 테이블을 참조하는 키 / 참조 무결성 유지










6. 무결성 제약조건

✔ 데이터 무결성(Integrity) 보장하는 제약 조건
1) 개체 무결성 : 기본키는 NULL 불가 & 중복 불가
2) 참조 무결성 : 외래키는 존재하는 값만 참조 가능
3) 도메인 무결성 : 컬럼의 데이터 타입, 범위 검사
4) 고유성 무결성 : 특정 컬럼 값이 중복되지 않도록

7. 슈퍼/서브 타입 데이터 모델

✔ 슈퍼/서브 타입 데이터 모델의 변환 기술
1) 개별로 발생되는 트랜잭션에 대해서는 개별 테이블로 구성
2) 슈퍼타입+서브타입에 대해 발생되는 트랜잭션에 대해서는 슈퍼타입+서브타입 테이블로     구성
3) 전체를 하나로 묶어 트랜잭션이 발생할 때는 하나의 테이블로 구성
